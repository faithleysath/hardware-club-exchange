# Product Requirements Document

## 1. Executive Summary

### Problem Statement

硬件社团成员之间存在稳定的二手交换需求，但当前交易信息主要散落在群聊、朋友圈或口头沟通中，导致发布信息易被淹没、历史记录难追踪、可信度和效率都不理想。

### Proposed Solution

建设一个部署在 Vercel 上、仅供社团内部使用的 Web 平台，支持成员登录、发布闲置、浏览筛选、预约沟通、状态更新与管理员审核。

### Success Criteria

- 平台上线后 30 天内，至少 60% 的活跃社团成员完成首次登录。
- 80% 的新发布闲置可在 24 小时内完成审核。
- 成员从新建发布到提交成功的中位时间不超过 3 分钟。
- 在 5,000 条闲置记录规模下，常见筛选请求服务端响应时间不超过 500ms。
- 所有管理员审核、下架、封禁动作都具备可追踪审计记录。

## 2. User Experience & Functionality

### User Personas

- 社团普通成员：浏览闲置、联系卖家、预约交易、收藏感兴趣物品。
- 社团卖家：发布闲置、管理图片与价格、处理预约申请、标记成交状态。
- 社团管理员：审核成员资格、审核闲置内容、处理举报、查看审计记录。

### User Stories

1. 作为社团成员，我希望可以通过 GitHub，或管理员预先创建的邮箱密码账号快速登录，这样我能顺利进入平台。
2. 作为社团成员，我希望能看到最新发布和可交易的闲置，这样我可以快速找到想要的设备。
3. 作为社团成员，我希望按分类、价格区间、成色和关键词筛选闲置，这样我可以缩小搜索范围。
4. 作为社团成员，我希望查看闲置详情和图片，这样我可以判断是否值得联系对方。
5. 作为社团卖家，我希望能发布闲置并上传多张图片，这样我能完整展示物品状态。
6. 作为社团卖家，我希望随时编辑价格、描述和联系方式，这样我可以保持信息准确。
7. 作为社团卖家，我希望把闲置标记为已预订、已成交或已下架，这样别人能看到真实状态。
8. 作为社团买家，我希望对某个闲置发起预约申请，这样我能表达购买意向而不是只靠群聊抢先。
9. 作为社团买家，我希望收藏感兴趣的闲置，这样我能稍后回看。
10. 作为社团成员，我希望举报明显违规、虚假或不合规的内容，这样平台能保持可信度。
11. 作为管理员，我希望审核新成员和新发布内容，这样平台能维持内部秩序。
12. 作为管理员，我希望查看关键操作日志，这样在发生纠纷时可以追溯处理过程。

### Acceptance Criteria

- 未经批准的用户不能看到平台内的闲置详情页。
- 普通成员只能编辑和下架自己发布的闲置。
- 每条闲置必须包含标题、分类、价格、成色、描述、至少一张图片和一种联系方式。
- 闲置在审核通过前不会出现在公开列表中。
- 预约申请必须记录申请人、目标闲置、申请时间和状态。
- 管理员下架或驳回任何内容时，系统必须记录原因。
- 搜索和筛选在移动端和桌面端都可用。
- 平台必须支持中文内容录入与展示。

### Non-Goals

- 不提供站内支付、担保、退款或资金托管。
- 不提供物流、快递或跨校配送流程。
- 不建设原生 iOS/Android App。
- 不实现复杂竞价、拍卖或拼单玩法。
- 不做开放给校外用户的公共二手市场。
- 不在第一版引入站内即时聊天系统。

## 3. AI System Requirements

本项目当前不包含 AI 功能，不设计模型调用或推理链路。

## 4. Technical Specifications

### Architecture Overview

平台采用单应用架构：

- Next.js 负责页面渲染、表单提交、服务端权限校验和管理台逻辑。
- Supabase Auth 负责登录、会话和成员身份基础能力。
- Supabase Postgres 存储业务数据。
- Supabase Storage 存储闲置图片。
- Drizzle 负责模式定义、迁移和类型化数据库访问。
- Vercel 负责托管与环境变量管理。

### Integration Points

- `Supabase Auth`：GitHub OAuth、管理员创建的邮箱密码登录、会话恢复、管理员权限绑定。
- `Supabase Postgres`：成员资料、闲置、预约、收藏、举报、审计日志。
- `Supabase Storage`：闲置图片、头像。
- `Vercel`：应用部署、预览环境、生产环境变量。

### Security & Privacy

- 平台默认内部可见，未通过资格审核的账号不得读取业务数据。
- 所有核心表启用 RLS，前端不可绕过权限。
- 管理员操作写入审计日志，确保可追责。
- 图片上传限制大小、类型与数量，避免滥用存储。
- 仅收集完成交易所需的最低限度个人信息。

## 5. Risks & Roadmap

### Phased Rollout

- MVP：成员登录、闲置浏览、详情、发布、图片上传、审核台、状态变更。
- v1.1：预约申请、收藏、举报、站内通知占位。
- v2.0：更强搜索能力、数据统计、自动归档与成员信誉增强。

### Technical Risks

- 如果成员入口限制设计不清晰，可能出现内部工具外泄或审核成本过高。
- 如果审核流程过重，管理员会成为上线后的主要瓶颈。
- 如果直接把交易沟通放在站内，复杂度会明显升高，拖慢第一版交付。
- 如果图片和详情页缺少规范，会让闲置信息质量参差不齐，影响平台口碑。
